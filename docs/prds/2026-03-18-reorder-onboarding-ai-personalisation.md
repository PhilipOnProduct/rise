## Overview

Reorder the five-step onboarding wizard so that preference collection (travel company, style tags, budget tier) moves to Step 3, before the AI activity preview in Step 4. Update the Claude prompt to use all three inputs explicitly. Ship the preference step, the prompt update, and the loading state as a single feature.

---

## Problem Statement

The AI activity preview currently fires with only destination and dates. The output is generic. Users who see generic output at the moment we're asking them to trust Rise will either disengage or conclude that AI travel apps are just like this.

Preferences are collected today but too late — after the AI has already made its first impression. The sequencing is backwards.

---

## User Need

Users need to feel understood before they commit to the product. The AI preview is the trust moment in onboarding. It needs to speak to a solo backpacker on a tight budget differently from a couple on a luxury city break — same destination, completely different trip. Right now it doesn't. Fixing this requires preferences before the preview, not after.

---

## Proposed Solution

**Reorder the wizard:**

| Step | Current | Revised |
|------|---------|---------|
| 1 | Destination | Destination |
| 2 | Dates | Dates |
| 3 | Hotel | Preferences (company, style, budget) |
| 4 | Activities preview | AI activity preview — using all three inputs |
| 5 | Account creation | Hotel |
| — | — | Account creation |

**Step 3 — Preference sweep.** Designed as a single above-the-fold screen on mobile. Three components, no scrolling:

- **Travel company** — single-select pill row: Solo / Partner / Friends / Family
- **Style tags** — multi-select grid, maximum 10 options, icon and label, 2-column layout. Options: Adventure, Food-led, Cultural, Nightlife, Relaxed, Off the beaten track, plus 3–4 others TBD with Maya
- **Budget tier** — three options only, descriptive labels with one-line concrete anchors: *"Street food and hostels" / "Mid-range hotels and sit-down meals" / "No compromises"*

No dropdowns. No sliders. The screen should take under 10 seconds to complete.

**Step 4 — AI activity preview.** The Claude prompt uses all three preference fields with explicit structure, not appended variables. Budget constraints are stated as instructions, not context. The loading state names the inputs back to the user: *"Finding activities for a solo trip, food-led, comfortable budget in Lisbon..."* This connection must be visible or Step 3 earns nothing.

**Preferences persist to the database.** They are not passed through in memory only. The itinerary row must hold `travel_company`, `style_tags`, and `budget_tier` for use in the day-view timeline, admin logs, and future recommendation scoring.

---

## User Stories

**As a user completing onboarding**, I want to tell Rise who I'm travelling with and what kind of trip I want before I see activity suggestions, so that the recommendations feel relevant to my actual situation rather than generic.

**As a user viewing the AI preview**, I want to see a loading state that reflects my inputs back to me, so that I understand my answers shaped the output and the step was worth completing.

**As a solo budget traveller**, I want activity suggestions that don't include luxury options I can't afford, so that Rise feels like it's working for me rather than ignoring what I told it.

**As a returning user**, I want my preferences saved to my itinerary, so that they carry through to the day-view timeline without me re-entering them.

## Technical Considerations

**Schema — add three columns in the current migration, not later:**

```sql
ALTER TABLE itineraries
  ADD COLUMN travel_company text,
  ADD COLUMN style_tags    text[],
  ADD COLUMN budget_tier   text;
```

Confirm before shipping that the wizard is writing to the itinerary row on completion, not holding state in component memory and discarding it.

**Claude prompt — structured, not appended.** Variables must shape the framing. Budget constraints require an explicit instruction:

```
You are recommending activities for a ${company} trip to ${destination}.
Budget tier: ${budget}. Style preferences: ${styleTags.join(', ')}.
Prioritise suggestions that match this profile specifically.
Do not recommend options that contradict the budget tier.
```

The final constraint is not optional. Without it, Claude will hedge and surface luxury options regardless of the budget selection.

**Wizard reordering.** Straightforward if steps are index-driven with no cross-step validation dependencies. Luca to confirm before estimating — URL routing or baked-in validation logic could add scope.

**Loading state.** Must reference the user's inputs explicitly. This is a UI component change in Step 4, not just a copy change.

## Risks & Open Questions

**Drop-off at Step 3.** This is the primary execution risk. A preference step that kills momentum before users see the payoff is worse than the current state. Maya's above-the-fold, fast-tap design is the mitigation. If the screen requires scrolling on mobile, cut components until it doesn't.

**The prompt update and loading state are not optional.** A new step that doesn't visibly change the AI output teaches users their answers don't matter. If either the structured prompt or the loading state slips, hold the release. These ship as a unit.

**Placement of budget tier — open for v2.** Alex raises a valid point: budget tier may land more naturally after hotel selection, when users have already made a spending signal. It's a plausible refinement. It adds wizard complexity now and delays the personalised preview. Revisit if Step 3 drop-off data supports it after launch.

**Style tag selection.** Final list of 8–10 options is not locked. Maya to confirm in design. Keep to a single screen regardless of final count.
