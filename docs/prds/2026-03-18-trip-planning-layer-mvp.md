# Rise Trip Planning Layer — PRD

**Author:** Sarah Chen, Product Manager
**Status:** Draft for team review
**Date:** 2025-07-14

---

## Overview

Rise is evolving its core experience from AI-powered concierge to structured trip planner. This PRD defines the MVP planning layer: a day-view itinerary surface that closes the gap between what our onboarding wizard promises and what the product currently delivers. This is not a full pivot. It is one well-scoped addition that either validates the planning mental model or tells us cheaply that it doesn't fit.

---

## Problem Statement

Our onboarding wizard asks users for destination, dates, and activity preferences. That sequence creates a clear expectation: Rise will help me plan my trip. Today, it doesn't. Users complete the wizard and receive recommendation streams with no structure, no sequence, and no place to organize what they find.

That mismatch is our fault. We manufactured an affordance and didn't fulfill it. The result is trust erosion that looks like low engagement but is actually a positioning failure. Users are mentally in planning mode when they arrive. We hand them a concierge.

---

## User Need

Travelers with upcoming trips need a single place to organize what they want to do, day by day. They don't need optimization or booking automation at this stage. They need structure: a way to move from "a list of things I might do" to "a rough plan for each day."

The core need is organization, not discovery. Rise already handles discovery. This layer handles what comes after.

---

## Proposed Solution

After onboarding completes, show users a **day-view timeline**. One column per trip day. Three time blocks per day: morning, afternoon, evening. AI pre-populates suggestions based on onboarding answers. Users can drag items between slots, dismiss suggestions, and add their own.

That's the full MVP scope. No map, no conflict detection, no booking integration, no collaborative editing.

The day-view screen is both a feature and a validation gate. If users engage — drag, reorder, dismiss — the mental model is confirmed and we build further. If they ignore it, we've learned something at low cost and we pull back.

**Build sequence:**

1. Lock and ship Luca's schema migration (the only irreversible decision)
2. Design and build the day-view timeline UI
3. AI pre-population from onboarding data
4. User controls: drag to reorder, dismiss, add manually

---

## User Stories

**As a traveler who just completed onboarding**, I want to see a rough day-by-day plan based on my stated interests so I have a starting point instead of a blank slate.

**As a traveler reviewing my itinerary**, I want to move activities between time slots so the plan reflects how I actually want to spend each day.

**As a traveler who finds a recommendation I like**, I want to add it to a specific day and time block so I don't lose it in a recommendation stream.

**As a traveler who doesn't like a suggestion**, I want to dismiss it without it reappearing so my itinerary stays relevant to me.

**As a returning user**, I want to open Rise and see my current trip plan without re-navigating to find it so the app feels like my home base for the trip.

---

## Success Metrics

**Primary validation metric:** 40%+ of users who reach the day-view screen perform at least one action (drag, dismiss, or manual add) within their first session. This is our signal that the planning mental model fits.

**Secondary metrics:**

- Day-view screen reach rate from onboarding completion (target: 80%+, measures whether we surface it clearly)
- Return visit rate for users with an active itinerary vs. without (measures whether planning creates retention)
- Items per itinerary at session end (measures engagement depth)
- Onboarding completion rate delta (we should not regress here)

**What we are not measuring yet:** booking conversion, revenue impact, collaborative features. Those are Q3 questions.

## Technical Considerations

Luca's schema is confirmed and unblocked. The core structure:

```
trips → itinerary_days → itinerary_items
              ↳ type: activity | restaurant | transport | note
              ↳ start_time, end_time
              ↳ location (PostGIS point)
              ↳ status: idea | confirmed | booked
              ↳ source: ai_generated | user_added | guide_tip
```

This is additive. It does not break existing onboarding schema. `itinerary_items` is the new surface.

**MVP scope at the data layer:** a structured checklist keyed by date and time block. Luca estimates two days of work on top of the existing schema sketch.

**Explicitly out of scope for this build:**

- Conflict detection and travel time between items
- Live availability from booking APIs
- Collaborative editing
- Map view (map is last, always — a sparse map erodes trust)

**Stack impact:** schema migration in Supabase, new UI components in Next.js/Tailwind, AI pre-population via existing Anthropic integration using onboarding data already in context.

---

## Risks & Open Questions

**Risk: Low day-view engagement.** If users don't interact with the timeline, we've confirmed the concierge model is the right frame and we revert cleanly. The schema migration is the only durable artifact — the UI is disposable.

**Risk: AI pre-population quality.** Poorly matched suggestions in the day-view will damage trust faster than an empty state. We need a minimum quality bar for pre-population before we ship. Empty slots with a prompt to add are better than bad AI suggestions.

**Risk: Scope creep on "planning."** Conflict detection, optimization, and booking integration will be requested immediately once a day-view exists. This PRD explicitly does not authorize those. They require a separate decision.

**Open question: Onboarding entry point.** Does the day-view replace the current post-onboarding state, or sit alongside it? Maya needs to resolve this in design before build begins.

**Open question: Behavioral signal from existing data.** Alex is pulling onboarding drop-off and activity selection data this week. If that data shows something surprising — low activity selection rates, high drop-off at the activities step — it should inform how aggressively we pre-populate the day-view. Review before we finalize AI prompt design.

**Open question: Session research.** We are running 2–3 observed sessions in parallel with build, not as a gate to it. If those sessions surface a fundamentally different mental model than what this PRD assumes, we have a decision point before we ship.
