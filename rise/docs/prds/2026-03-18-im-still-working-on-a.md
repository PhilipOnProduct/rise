# Rise Onboarding — Product Requirements Document

**Author:** Sarah Chen, Product Manager
**Status:** Draft v1.0
**Last Updated:** [Today's Date]

---

## Overview

This PRD defines the first-version onboarding experience for Rise. The scope is narrow and intentional: a three-step wizard that takes a first-time visitor from zero context to a personalised AI-generated travel plan, with account creation gated at the end. No payments, no booking integration, no advanced personalisation. Just a fast, impressive first session that earns a signup.

---

## Problem Statement

We don't yet know if Rise can deliver enough value, fast enough, to earn a user's trust before they leave.

First-time visitors arrive with no context about what Rise is or why it's better than Googling. A blank dashboard will cause them to bounce immediately. We need to prove value during the first session — before we ask for any commitment.

The existing 5-step wizard (destination → dates → hotel → activities → account) is structurally reasonable but over-engineered for an MVP with no users. It asks for too much before delivering anything. The order also needs to change: account creation must come last, after the user has already experienced something worth saving.

---

## User Need

> "Help me feel like my trip is being taken care of — quickly and effortlessly."

The user isn't looking for a form. They're looking for a feeling: someone competent is handling this. Every design and engineering decision in this build should serve that feeling.

---

## Proposed Solution

A three-step onboarding wizard that feels like a concierge conversation, not a signup flow. Wizard state is persisted in `localStorage` so progress isn't lost. AI generation fires once — immediately after account creation — so the user lands on a dashboard that already has something intelligent in it.

**Step 1 — Destination + Dates**
One screen. Full-screen cinematic input. Single question: *"Where are you going?"* Autocomplete with a hero image or gradient that shifts per destination. Dates inline on the same screen. No progress bar at this stage — don't signal a long form before the user has typed a single character.

**Step 2 — Hotel**
Optional single-field input for hotel name or area. As the user fills this in, a visible plan card begins assembling on screen — destination badge populates, dates slot in, hotel appears. This is the micro-moment that shifts the mental model from "form" to "concierge building something for me." Progress indicator appears here for the first time, framed as low-commitment.

**Step 3 — Account Creation**
Clear value hook: *"Save your plan and get your AI recommendations."* Email and password only. No social auth for v1. On submit, AI generation begins immediately. The user lands on a dashboard where a personalised plan summary is already streaming in.

**What's out of scope for v1:**
Activities step (removed — adds cognitive load before we've validated core value). Mid-flow AI streaming (deferred — latency and error states inside the wizard are too risky). Social login. Booking integration.

## User Stories

**As a first-time visitor,** I want to enter my destination and dates quickly so I can see what Rise will do for me before committing to anything.

**As a first-time visitor,** I want to see my plan taking shape as I fill in details so I understand that Rise is building something for me, not just collecting data.

**As a first-time visitor,** I want a clear reason to create an account so that signing up feels like a reward, not a gate.

**As a new user,** I want to land on a dashboard that already has personalised content so I immediately understand the value of the product I just signed up for.

**As a returning visitor who dropped off mid-wizard,** I want my progress to still be there so I don't have to start over.

---

## Success Metrics

**Primary metric:** Wizard completion rate — percentage of visitors who reach and submit the account creation step. Target: ≥ 40% of users who start Step 1.

**Secondary metrics:**
- Step drop-off rate per step (flag any step above 30% drop-off for immediate investigation)
- Time to complete wizard (target: median under 3 minutes)
- Dashboard engagement after signup — did the user interact with their AI plan? (click, scroll, or dwell > 30 seconds counts)

**Qualitative signal:** In observed sessions with five known users, do they comment on the AI output positively without prompting? This is our gut-check before any growth push.

**What we're explicitly not measuring yet:** Retention, repeat visits, or revenue. This sprint is about proving we can earn a first account. Everything else comes after.

## Technical Considerations

**Wizard state:** Persist in `localStorage` for MVP. Not elegant, but zero cost, zero complexity, and sufficient for our user volume. Migrate to Supabase session-linked drafts in a later sprint if anonymous-to-auth recovery becomes a priority.

**AI generation:** Fires once, post-signup, with full wizard context passed to the Anthropic API. Do not trigger mid-wizard. Streaming renders on the dashboard, not inside the wizard flow — this keeps the wizard fast and error-free, and makes the dashboard feel alive on first load.

**Prompt quality is a prerequisite, not a sprint 2 item.** The Anthropic prompt for the plan summary must be written, tested, and passing a team gut-check across at least three different destinations before wizard development begins. A generic output ("Try pasta in Rome") will actively destroy trust at the most important moment in the product. This is the highest-risk item in the build.

**Auth:** Supabase email/password only. No social providers for v1 — keep the surface area small.

**Stack alignment:** Next.js 16 for wizard steps as client components, Supabase for auth and user record creation, Anthropic API with streaming for plan generation, Vercel for deployment. No new infrastructure required.

## Risks & Open Questions

**Risk: AI output quality.** This is the single highest-risk item. A mediocre first plan summary will damage trust more than no AI at all. Mitigation: prompt development and internal review before any user-facing build ships. Do not skip this step.

**Risk: Wizard feels slow or broken on poor network.** The wizard itself contains no AI calls, which reduces this risk significantly. The streaming dashboard load should have a clear loading state with a message like *"Your plan is being prepared"* so latency feels intentional, not broken.

**Risk: We don't know our acquisition channel yet.** The wizard is designed for a user arriving with trip-planning intent. If our primary channel delivers users with different intent (e.g., "find a restaurant"), the wizard metaphor may not fit. Mitigation: watch session recordings closely in early access and be prepared to revisit Step 1 framing.

**Open question: What happens if the user skips the hotel field?** We need a fallback prompt path that generates a compelling plan summary with only destination and dates. This needs to be tested explicitly — it may be the most common input pattern.

**Open question: Do we offer a "not booked yet" path for hotel?** Maya's design flagged this. It's worth handling gracefully in copy even if the underlying logic is the same.

**Open question: Activities step — when does it come back?** Removed from v1 to reduce cognitive load. Should be revisited once we have completion rate data and understand whether users feel the plan summary is specific enough without it.
